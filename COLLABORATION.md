# Collaboration Guidelines / 协作制度

> "我们不看资历，不看职称，只看 GitHub 的实际贡献。AI 是主力航海士，掌握物理直觉的你，是唯一的舵手。"

本文档是 OpenSCI-Ocean 项目的协作总纲，所有参与者在加入前请仔细阅读并认同以下条款。

---

## 1. 项目性质 / Project Nature

这是一次**公开透明的探索性实验**，不是一个发表承诺。我们不知道最终能不能发出来，但我们会认真地试。

- 目标期刊：Nature Geoscience、Nature Climate Change、Nature Communications 等 Nature 子刊
- 全过程公开记录，包括失败、错误和被拒
- AI（Claude、GPT 等大模型）作为核心生产工具，人类负责审核、纠偏和物理判断
- 投稿时将在 Methods 或 Acknowledgements 中如实披露 AI 的参与方式和程度

## 2. 角色定义 / Roles

### 2.1 项目负责人 / Project Lead
- 负责项目整体方向、里程碑推进和最终投稿决策
- 拥有论文署名顺序的最终裁定权（在充分协商的基础上）
- 管理 GitHub 仓库的 merge 权限

### 2.2 Prompt Engineer / AI 交互负责人
- 将科研目标拆解为 AI 可执行的指令链（Prompt Stack）
- 负责数据预处理、AI 代码生成、初稿框架搭建
- 记录所有关键的 AI 对话日志（Prompt Logs）

### 2.3 Domain Expert / 物理直觉把关人
- 在关键节点进行"物理熔断"——判断 AI 的结论是否在物理上成立
- 重点关注：谱分析斜率、动力学解释、量级估算、因果推断
- 不要求每天参与，但在关键评审节点必须响应

### 2.4 Code Reviewer / 代码审查员
- 验证 AI 生成的代码是否存在逻辑漏洞
- 确保代码能高效处理大尺度数据（SWOT、CMEMS 等）
- 审查数据处理流程的可复现性

### 2.5 Literature Scout / 文献侦察兵
- 跟踪前沿文献，确认选题新颖性
- 对 AI 生成的每一条参考文献进行真实性核验
- 找到合适的对标论文，评估投稿可行性

> 一个人可以同时担任多个角色。角色可以在项目进展中动态调整。

## 3. 参与者要求与加入方式 / Participant Requirements & How to Join

### 3.1 基本要求

本项目欢迎所有对海洋科学研究感兴趣的参与者。由于 AI 承担了大部分技术执行工作，我们对参与者的要求侧重于**科学判断能力**而非特定技术栈：

| 要求 | 说明 |
|---|---|
| 学科背景 | 自然科学相关专业（物理、数学、地学、海洋、大气、遥感、环境、生态、计算机等均可）。非自然科学背景的参与者若能证明具备相关知识储备，同样欢迎 |
| 学位要求 | **不设学位门槛。** 本科在读、硕博研究生、博士后、教职人员、独立研究者、行业从业者均可参与。关键是能看懂英文论文、能对研究结论作出独立判断 |
| 英文能力 | 能阅读和理解英文学术论文。论文写作由 AI 辅助完成，但审稿回复和与编辑的通信需要基本的英文写作能力 |
| 工具能力 | 能使用 GitHub 进行基本的协作操作（Fork、Branch、PR、Issue）。不熟悉的参与者可参考 [CONTRIBUTING.md](CONTRIBUTING.md) 快速上手 |
| 时间投入 | 不要求全职参与。但加入某个项目后，需要在关键节点（D2 审查、D3 评审）及时响应，承诺响应时间不超过 48 小时 |

### 3.2 不同角色的额外要求

| 角色 | 额外要求 |
|---|---|
| Domain Expert / 物理把关人 | 需具备相关研究方向的专业知识，能判断物理量级是否合理、因果推断是否成立。通常为相关方向的研究生及以上 |
| Code Reviewer / 代码审查员 | 熟悉 Python 数据分析生态（NumPy、xarray、Matplotlib 等），能看懂和调试数据处理代码 |
| Literature Scout / 文献侦察兵 | 能高效使用 Google Scholar、Web of Science 等文献检索工具，具备识别文献真伪的能力 |
| Prompt Engineer / AI 交互 | 有使用 Claude / GPT 等大模型的经验，能将科研问题拆解为有效的 Prompt 指令 |

### 3.3 如何加入

1. **阅读本文档和 [README.md](README.md)**，了解项目性质和协作规则
2. **选择感兴趣的项目**，查看 [projects/DASHBOARD.md](projects/DASHBOARD.md) 了解各项目状态
3. **通过以下任一方式联系**：
   - 在 GitHub 上 [提交 Issue](https://github.com/pangeo-data/OpenSCI-Ocean/issues)，标题格式：`[加入] 你的姓名 - 感兴趣的项目编号`
   - 直接向感兴趣的项目提交 PR（如文献补充、代码审查、分析改进）
   - 通过微信群联系（见 [招募帖](docs/recruitment_cn.md)）
4. **项目负责人确认后**，在 CONTRIBUTORS.md 中登记，正式加入项目

### 3.4 试用期

新参与者加入后有一个自然的磨合期（通常为一个完整的项目阶段，如 D0 或 D1）。磨合期内：
- 双方相互了解协作风格和能力水平
- 参与者可以随时无条件退出（参见第 9 节退出机制）
- 若项目负责人认为参与者的贡献与项目需求不匹配，会坦诚沟通，协商调整角色或退出

## 4. 署名制度 / Authorship Policy

署名是学术项目最敏感的问题。我们采用**贡献导向、透明量化、前置协商**的原则。

### 3.1 署名标准

遵循 [CRediT（Contributor Roles Taxonomy）](https://credit.niso.org/) 国际学术贡献分类标准。署名要求满足以下**至少两项**：

| CRediT 维度 | 说明 |
|---|---|
| Conceptualization | 提出核心科学问题或研究假设 |
| Methodology | 设计研究方案或分析方法 |
| Software | 编写或审查关键代码 |
| Validation | 物理机制验证与纠错 |
| Formal Analysis | 数据分析与解读 |
| Data Curation | 提供或整理关键数据 |
| Writing – Original Draft | 撰写论文初稿 |
| Writing – Review & Editing | 实质性修改论文内容 |
| Visualization | 制作关键科学图表 |
| Supervision | 项目监督与方向把控 |

### 3.2 署名顺位规则

#### 第一作者

第一作者授予对论文贡献最大的参与者。具体判定依据（按权重排序）：

1. 提出核心科学问题并在 AI 辅助研究中起决定性纠偏作用
2. 主导了 70% 以上的 Prompt 交互、数据分析流程搭建和论文框架设计
3. 完成了关键的物理机制验证和结果解读

若两人贡献难以区分，可设**共同第一作者**（Co-first author），在论文脚注中标注"These authors contributed equally"。共同第一的排序由双方协商决定，协商不成则由项目负责人裁定。

#### 通讯作者

通讯作者是论文投稿、审稿回复和学术质量的最终责任人。通常为以下之一：

- 项目发起人（Initiator），即 OpenSCI-Ocean 仓库的创建者和维护者
- 提供核心未发表数据的 PI（如提供安海浮标数据的负责人）
- 具有独立 PI 资格、愿意为论文学术质量背书的资深研究者

通讯作者需要承担的具体责任：
- 审核论文最终版本，确保科学结论可靠
- 负责与期刊编辑的所有通信（投稿、审稿回复、校样确认）
- 论文发表后负责回应学术质疑
- 管理原始数据的存档和共享（按期刊要求）

若第一作者本身具有独立 PI 资格，可以同时担任通讯作者。

#### 共同作者

满足署名标准（3.1 中至少两项）且有实质性 GitHub 贡献记录（Issue 讨论 / PR / Commit）的参与者。

#### 致谢（非署名贡献）

以下贡献不满足署名标准，在 Acknowledgements 中致谢：
- 仅提供数据访问权限但未参与分析
- 提供一般性意见或建议但未实质参与研究
- 提供计算资源或基础设施支持

#### 署名协商时间节点

- **D0 阶段结束时**：初步确定拟定作者列表和预期顺序，在项目 README 中记录
- **D2 阶段结束时**：根据实际贡献调整作者列表，全体参与者确认
- **D4 投稿前**：最终确认署名顺序，所有作者签字认可（GitHub Issue 中留下书面同意记录）

署名顺序一经投稿确认，原则上不再变更，除非审稿过程中有新的实质性贡献产生。

### 3.3 贡献追踪

所有贡献必须在 GitHub 上留下可追溯的记录：

- 使用 **CONTRIBUTORS.md** 动态维护贡献看板
- 每个里程碑（Milestone）结束时，集体评审贡献记录
- 署名争议由全体参与者协商解决，项目负责人拥有最终裁定权

### 3.4 AI 的署名问题

AI（Claude、GPT 等）**不列为作者**。原因：
- AI 不能对研究内容承担学术责任
- Nature 系列期刊明确要求 AI 不能作为作者
- AI 的贡献将在 Methods 部分如实声明

## 5. 项目结构与工作流 / Project Structure & Workflow

### 4.1 多项目并行

本仓库包含 **10 个独立研究项目**（P01–P10），每个项目是一个完整的科学探索单元：
- 每个项目有独立的科学问题、数据、代码和论文
- 项目之间互不依赖，可以独立推进
- 参与者可以加入一个或多个项目
- 署名按项目独立计算

### 4.2 项目生命周期

每个项目遵循快速迭代节奏：

```
D0 (< 1天)    D1 (1天)       D2 (2-3天)       D3 (~3天)       D4 (1天)
选题确认   → AI 初稿生成 → 人类审查交付 → 内部评审定稿 → 投稿
```

| 阶段 | 代号 | 内容 | 产出 |
|---|---|---|---|
| D0: Explore | 选题 | 文献快速调研、判断新颖性、锁定科学问题 | 项目 README 填写完成 |
| D1: Draft | 初稿 | AI 全流程生成（文献综述 + 数据分析 + 图表 + 全文） | 初稿 + 代码 + 图表 |
| D2: Deliver | 交付 | 人类审查纠偏、物理把关、打磨交付稿 | 交付稿 |
| D3: Review | 审查 | 模拟审稿、交叉评审、查漏补缺 | 定稿 |
| D4: Submit | 投稿 | 格式检查、Cover Letter、正式投稿 | 投稿确认 |

### 4.3 GitHub 协作规范

**GitHub 是所有协作的中心平台。** 不同用户之间通过 GitHub 进行协作。

#### Commit 规范

关键 commit 是贡献的核心凭证。Commit message 格式：

```
[P01] D1: AI 生成 SWOT SSH 谱分析初稿

- 完成文献综述（32 篇核心文献）
- SWOT L3 数据波数谱计算代码
- 图 1-4 生成
```

格式：`[P##] D#: 简要描述`

#### Branch 规范

- 每个项目使用独立分支：`p01/main`、`p02/main`
- 功能分支：`p01/feature/spectral-analysis`
- 合入 `main` 前需要 PR Review

#### Issue 规范

- Issue 标题格式：`[P01] 简要描述`
- 使用 Label 标记项目编号和阶段（`P01`、`D0-explore`、`D1-draft` 等）
- 关键科学决策必须有 Issue 记录

#### PR 规范

- 每个阶段完成后提交 PR
- PR 描述中列出本阶段的关键产出和变更
- PR 合并需要至少 1 位参与者的 Review

### 4.4 沟通方式

- **GitHub Issue / PR**：所有技术讨论和决策的唯一官方记录（Single Source of Truth）
- **微信工作群**：日常沟通、快速对齐
- 微信中的关键决策必须同步到 GitHub Issue

## 6. AI 使用规范 / AI Usage Policy

### 5.1 透明记录

- 所有关键的 AI 对话必须保存 Prompt Log，存放在各项目的 `logs/` 目录
- 记录格式：日期 + 使用的模型 + 任务描述 + 关键输出摘要
- 投稿前汇总 AI 参与度报告

### 5.1b 论文全过程记录

GitHub 完整保存论文从零到投稿的全过程，每个项目目录包含：

```
projects/p01/
├── README.md          # 项目卡片：科学问题、假设、状态
├── literature/        # AI 文献调研素材、文献笔记
├── analysis/          # 数据分析脚本（不含原始数据）、数据源链接
├── figures/           # 所有生成的图表（含中间版本）
├── manuscript/
│   ├── v1_ai_draft/   # D1: AI 生成的初稿
│   ├── v2_delivery/   # D2: 人类审查后的交付稿
│   ├── v3_final/      # D3: 内部评审后的定稿
│   └── submitted/     # D4: 最终投稿版本 + Cover Letter
└── logs/              # AI 交互日志
```

论文的每一版都保留，通过 Git 历史可以完整追溯从 AI 初稿到人类终稿的每一步修改。

### 5.2 质量红线

- **文献幻觉零容忍**：AI 生成的每一条参考文献，必须通过 Google Scholar / Crossref 进行人工核验。发现编造文献的 Commit 直接 Revert
- **物理量级检查**：AI 给出的任何数值结果，必须由 Domain Expert 进行量级合理性检查
- **代码可复现**：AI 生成的所有代码必须能在标准环境中复现

### 5.3 数据安全

- 安海浮标的未发表现场观测数据，使用 API 时必须确认数据不用于模型训练（Data Opt-Out）
- 敏感数据不得直接粘贴到公共 AI 对话界面
- 数据文件不上传到公开仓库，通过 `.gitignore` 排除

### 5.4 计算资源

数据处理和分析在参与者的本地环境或租用的远程服务器上完成，不依赖 GitHub 的计算能力。

- **本地计算**：参与者在个人电脑上运行分析脚本
- **远程服务器**：需要大规模计算时，租用云服务器（费用由使用者自行承担，或项目组协商分担）
- **GitHub 只存代码和结果**：分析脚本上传到项目的 `analysis/` 目录，关键结果（图表、统计数据）上传到 `figures/`，原始数据和中间数据不上传
- 脚本中应包含清晰的环境配置说明（`requirements.txt` 或 `environment.yml`），确保任何人可以在自己的环境中复现

## 7. 数据管理 / Data Policy

**核心原则：GitHub 仓库不存储任何原始数据文件，只保存数据的链接地址和下载脚本。**

### 6.1 数据存储规则

- **禁止上传**：原始数据文件（.nc、.hdf5、.mat、.csv 等大文件）
- **应该上传**：数据下载脚本、数据源 URL/DOI、数据说明文档
- 每个项目的 `analysis/` 目录中包含数据获取脚本，任何人可以复现数据下载

### 6.2 公开数据

公开数据源（CMEMS、ERA5、Argo、SWOT L3/L4 等）：
- 在项目 README 中列出数据源链接和版本号
- 提供完整的下载脚本（Python/Shell），确保可复现

### 6.3 私有数据

青岛安海 GNSS 浮标现场观测数据属于项目独家资源：
- 数据所有权归青岛安海信息技术有限公司
- 仅限项目内部使用，未经数据提供方书面同意不得外传
- GitHub 上只记录数据描述和元信息，不上传原始数据
- 论文发表后，根据期刊要求决定是否在公开数据仓库（如 Zenodo）存档

### 6.4 数据引用

所有使用的数据集必须在论文中正确引用，包括 DOI 和版本号。

## 9. 退出机制 / Exit Policy

参与者可以在项目任何阶段退出，无需说明理由。但不同阶段退出的处理方式不同：

### 8.1 主动退出

| 退出时间 | 署名处理 | 工作交接要求 |
|---|---|---|
| D0 阶段（选题调研） | 不署名，在 Acknowledgements 中致谢调研贡献 | 将已有调研材料整理到项目 `literature/` 目录 |
| D1 阶段（AI 初稿） | 根据实际贡献判断是否署名；若已有不可替代的实质贡献（如核心 idea），保留署名权 | 提交当前工作的 PR 或在 Issue 中记录进展 |
| D2–D3 阶段（审查/定稿） | 保留署名权，署名顺序根据实际贡献调整 | 完成手头审查意见的书面记录 |
| D4 阶段（已投稿） | 署名不变 | 仍需配合审稿回复中涉及自身贡献部分的问题 |

### 8.2 退出的权利与义务

- **权利**：退出者保留其已完成贡献的署名权，任何人不得在退出后擅自移除其署名
- **义务**：退出时将手头未完成的工作以文档形式交接（Issue 或 PR），确保后续参与者可以接手
- **代码和记录**：退出者在项目中的代码提交、Issue 讨论、PR 记录等公开贡献记录保留不动
- **数据**：若退出者提供了私有数据，需明确该数据是否仍授权项目使用。若撤回数据授权，项目需删除相关数据并调整分析方案

### 8.3 失联处理

若参与者在关键阶段（D2–D4）连续 14 天未回应任何联系方式（GitHub / 微信），且未事先请假：
- 项目负责人在 GitHub Issue 中公开记录联系尝试
- 再等待 7 天后，视为默认退出，按 8.1 的规则处理
- 若后续恢复联系，署名权仍根据实际贡献保留

### 8.4 项目整体终止

若某个项目因科学原因（假说被证伪、数据不可用等）决定终止：
- 在项目 README 中标注终止状态和原因
- 所有已产出的代码、图表、分析记录保留在仓库中（作为公开实验记录的一部分）
- 参与者可自由将自己的贡献用于其他研究（遵守 MIT License）

## 10. 投稿策略与拒稿处理 / Submission Strategy & Rejection Policy

### 9.1 目标期刊分级

每个项目在 D0 阶段确定目标期刊，采用**三级投稿策略**：

| 级别 | 说明 | 典型期刊 |
|---|---|---|
| T1（首投） | 领域顶刊，影响因子最高 | Nature Geoscience, Nature Climate Change |
| T2（改投） | 高水平综合/专业期刊 | Nature Communications, Science Advances, GRL |
| T3（保底） | 领域主流 SCI 期刊 | JGR-Oceans, JPO, Remote Sensing of Environment |

### 9.2 拒稿后的改投流程

拒稿是学术出版的正常环节，本项目对拒稿的处理原则是：**不放弃有价值的工作，但也不无限降级。**

1. **T1 拒稿 → 改投 T2**
   - 分析审稿意见，判断是"水平不够"还是"方向不合适"
   - 若审稿意见有建设性修改建议，修改后改投 T2
   - 修改周期不超过 2 周，在 GitHub Issue 中记录审稿意见和修改方案
   - 改投时需根据 T2 期刊的格式和风格要求调整论文

2. **T2 拒稿 → 改投 T3**
   - 同样分析审稿意见并修改
   - 若两次拒稿的审稿意见指向相同的根本性问题（如数据不充分、方法有缺陷），需认真评估论文是否值得继续投稿
   - 项目全体作者讨论决定是否继续改投

3. **T3 拒稿 → 项目作者集体讨论**
   - 三级投稿均被拒可能意味着研究本身存在根本问题
   - 选项：(a) 大幅修改后重新投稿；(b) 作为预印本发布在 ESSOAr / arXiv；(c) 终止该项目
   - 决定在 GitHub Issue 中记录

### 9.3 审稿回复

- 审稿回复由通讯作者负责协调，所有作者参与
- 审稿意见和逐条回复在 GitHub Issue 中存档
- 修改稿的 diff 通过 PR 提交，确保修改过程可追溯

### 9.4 预印本政策

- 在正式投稿前或投稿同时，鼓励将论文提交到预印本服务器（ESSOAr、arXiv 等）
- 预印本可以建立时间优先权，对开放科学实验有额外意义
- 是否发预印本由全体作者协商决定，需全体同意

## 11. 费用分担 / Cost Sharing

### 10.1 版面费（Article Processing Charge, APC）

开放获取期刊通常收取版面费。不同期刊的 APC 差异很大：

| 期刊 | 大致 APC（参考值） |
|---|---|
| Nature Communications | ~$5,690 / ~¥41,000 |
| Nature Climate Change / Nature Geoscience | ~$11,390 / ~¥82,000（OA 选项） |
| GRL | ~$3,500 / ~¥25,000 |
| JGR-Oceans | ~$3,000 / ~¥22,000 |
| Remote Sensing of Environment | ~$3,390 / ~¥24,000 |

> 以上为 2025 年参考价格，实际以投稿时期刊官网为准。部分期刊对发展中国家作者有减免政策。

#### 版面费分担原则

1. **优先由通讯作者所在单位的科研经费支付**——这是学术界的通行做法，通讯作者对论文承担学术责任，其单位通常有对应的发表经费
2. **若通讯作者单位无法承担**，由全体作者按贡献比例协商分担
3. **若全体作者均无经费来源**，优先选择非 OA 发表途径（部分期刊允许传统订阅模式，无 APC），或申请期刊的 APC 减免/豁免（waiver）
4. 版面费分担方案必须在**投稿前**确定，并在 GitHub Issue 中书面记录，避免论文接收后产生费用纠纷

### 10.2 其他费用

| 费用项目 | 承担方式 |
|---|---|
| AI API 调用费（Claude/GPT） | 各参与者自行承担各自使用的部分 |
| 云计算/服务器费 | 使用者自行承担，或项目组协商分担 |
| 数据获取费（如有） | 由需要该数据的项目参与者协商分担 |
| 英文润色/翻译费（如有） | 由全体作者协商分担 |

### 10.3 无偿参与原则

本项目不设任何报酬或劳务费。所有参与者均为志愿参与，唯一的"回报"是署名权和公开的贡献记录。参与者应在加入前充分了解这一点。

## 12. 争议解决 / Dispute Resolution

1. 技术争议：在 GitHub Issue 中公开讨论，以证据和数据为准
2. 署名争议：全体参与者协商，项目负责人最终裁定
3. 如无法达成一致，可邀请项目外的资深学者作为中立仲裁

---

## 13. 修订 / Amendments

本制度可以根据项目进展修订。修订需要：
- 在 GitHub Issue 中提出修改提案
- 至少 2/3 活跃参与者同意
- 通过 PR 合并更新
